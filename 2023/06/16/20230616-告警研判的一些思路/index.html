告警研判的一些思路

发表于2023-06-16|更新于2023-06-19
|字数总计:3.8k|阅读时长:11分钟|阅读量:

暴力破解

  • 看攻击ip来源

    • 内网ip

      • 看目的ip覆盖面，如同一源ip爆破同一网段的多台主机

        • 询问客户是否为内部扫描导致等，例如内部的扫描器(可记录客户的扫描机ip)

          • 建议客户在hw或重保期间，不要进行内部扫描，或扫描前通知(需精确到扫描时间及扫描范围)

      • 同时看爆破次数及详情，一般来说如果为批量内网爆破，告警主机的告警次数及详情会较为相似

      • 看爆破时间，是否为凌晨或就餐这种非正常工作时间

        • 和客户确认

    • 外网ip

      • 一般来说只要不是弱口令，不会有太大危害，根据客户要求决定是否封禁

        • 封禁有两种方式，都需跟客户进行确认是否封停

          • 方式一：直接爆破告警后直接点封停，建议在未自动封停时先手动点封停

          • 方式二：手动用微隔离快速任务封禁得跟客户确认一下是否需要用微隔离快速任务进行封禁(该操作会在机器的防火墙iptables上添加相关drop规则)

  • 看告警是否跟微蜜罐告警同时出现

    • 客户扫描机扫描以及其他漏扫情景会出现爆破与蜜罐端口扫描同时出现，这种告警特征非常明显，最好是跟客户进行确认是否是内部漏扫

异常登录

  • 异常登陆需要先配置正常登录规则才会有告警，所以当出现告警时，可以直接询问客户是否为自己操作，可疑概率较大（天翼云这边由于资产数量太多没有开启该项功能）

反弹shell

  • 查看告警详情进程树

    • 需要了解常见反弹shell姿势（​反弹shell常见姿势（持续更新）​）

      • 没见过的命令行，不确认是否恶意的，自行百度，如确认为异常操作，需跟相关安全员进行确认

    • 主要查看进程树关系以及命令行

      • 如进程树最后几项是由web中间件起的，后面是反弹shell的命令及进程，大概率有问题，检查告警ip的”漏洞检测”结果，查看是否有高危rce漏洞，或检查是否有webshell

        • 如果反弹到的ip为内网ip，则表示该内网主机可能也失陷了，需要一同排查

      • 存在一定程度的误报，判断是不是误报主要看进程树中倒数几个进程的命令行内容，如果内容看着就是执行的业务脚本之类，虽然最后确实连通了某个ip，但是这种就是误报，需要结合业务来判断

本地提权

  • 查看告警详情进程树

    • 关键要看最后一个告警进程的命令行

      • 同时查看运行用户的变化，如果为一个web用户、数据库用户、应用用户等非root用户变为了root用户，则可疑性较大

    • 需要对常见提权方式进行了解（​Linux操作系统提权​）

      • 无法确认是否为提权，百度是否有这种提权方式

后门检测

  • 首先要明确告警类型

    • 可疑进程参数

      • 主要是基于一些常用攻击命令中携带的参数做匹配来告警，例如frp等

        • Linux

          • 有些明确写着是哪种恶意程序的参数，可以直接定位，相对好判断

            • 查看检测说明，确认是哪种问题

            • 查看静态信息

              • 查看文件所属用户、用户组

                • 根据文件系统的权限继承原则，可以辅助确认是哪个应用或用户被攻破，例如所属用户是redis，则可能是通过redis未授权访问攻击的，所属用户是liming，则liming的密码大概率已丢失

                  • 如果为密码丢失引起，检查对应失陷用户的操作日志

            • 查看文件修改时间（ctime）

              • 只有当对文件进行赋权等操作时，会改变ctime，根据ctime可大致判断该文件的落地时间，大概率为攻击者攻入的时间

            • 查看进程树

              • 查看上方父进程，是否是web中间件之类，可能通过rce漏洞打入后写入

              • 如果上方父进程为系统进程，则需要进一步判断，如crontab，是计划任务启动，可能是攻击者建立的计划任务后门，父进程为systemd启动，则有可能为通过ssh连入，之后上传运行的

        • Windows

          • 情况相比Linux来说要更加复杂一些，需要一些专业知识来作为辅助

            • 可疑进程参数这里有些和Linux情况一样，涉及一些比较明显的工具利用，包括att&ck中的一些组件利用等，还有涉及注册表等的一些操作。这种情况一般攻击者已经拿到了主机权限了，需要通过多种告警来结合判断。

              • 还有一种情况，是涉及rundll32.exe程序的可疑进程参数告警，这种目前观察到两种情况

                • 一种是与web命令执行类型，前方进程为w3wp.exe紧接着起了个rundll32.exe的告警。这种比明显，就是web漏洞直接通过cs上线了，排查方式与web命令执行类似。

                • 另一种为单独的rundll32.exe的可疑参数告警，这种的排查方式按照黄浩宇的说法为通过工具箱中的dump内存功能，dump该告警进程的内存，上传至威胁情报平台进行排查。

    • 隐藏进程

      • 攻击者常用的一种隐藏后门的方式，存在一定误报

        • 查看静态信息

          • 查看文件所属用户、用户组，判断文件是哪个用户传入的

            • 对该用户操作记录或该用户对应应用进行排查，与上方一致

        • 查看文件位置

          • 有些可以通过存放位置大致判断出是否为恶意文件，常见的如/tmp，/var/tmp等

        • 查看修改时间

          • 大致判断攻入时间

        • 查看进程树

          • 大致判断该文件落地方式，和上面一样

        • 查看文件类型

          • Linux下通过file命令查看该文件类型，之后方便进行进一步判断，若出现文件类型与后缀名不一致的情况，大概率为恶意文件

            • 后缀名为Java，但文件类型为Linux可执行文件，异常

        • 上传到微步或virustotal中进行判别

    • 应用后门

      • 攻击者通过替换系统默认程序留后门的一种方式，万相通过hash比对，有一定的误报

        • 一般是因为升级软件之后没有更新rpm缓存导致，可以把告警中的sha256值在威胁情报网站进行查询判别

          • 如果是真实后门文件

            • 查看静态信息，查看传入时间，查看文件所属用户、用户组，判断文件是哪个用户传入的，对该用户操作记录或该用户对应应用进行排查

          • 如果确认不是后门，进行加白

    • Rootkit

      • 一般告警内容是发现与某已知rootkit相同特征文件

        • 下载该文件上传至威胁情报网站进行判别，或通过sha256值进行查询，可能存在一定误报

          • 确认是恶意文件

            • 查看静态信息进一步排查关联，和上面方法一样

    • 可疑脚本内容

      • 通过规则检查脚本内容来进行的判断，存在一定误报

        • 确认为恶意时，需要对脚本内容进行判断，查看静态信息判断入口等，与上面类似

    • HackTool、Trojan等类

      • 这种有点类似于杀毒的功能，基本都可以明确检测出具体的工具或后门类型，如msf、一些隧道工具等。

        • 排查方式也是通过静态信息的时间来判断文件落地时间，通过进程信息判断启动方式，来判断是计划任务，还是从桌面双击启动等。

    • 可疑模块

      • 可疑模块这个conf上的资料比较少，且目前误报较高，不是非常好判断

    • 其他

      • 通用方法

        • 上传恶意文件至威胁情报平台进行研判

          • 确认恶意

            • 查看静态信息，对应文件用户，上传时间，进程，判断入口点

        • 使用恶意文件对应的sha256值进行研判

          • 同上

web后门

  • 一般是指webshell等动态脚本类恶意文件，存在一定误报

    • 通过万相检测说明进行大致判断后面类型

      • 代码执行

        • 不管哪种类型告警，确认为攻击后

          • jsp、jspx 等

            • 通过反序列化漏洞攻入

              • 通过安全风险—漏洞检测模块查看告警主机是否有例如shiro、fastjson等其他存在rce漏洞的组件

            • 通过web的文件上传功能

              • 若无相应漏洞组件，则大概率通过上传功能直接上传，需要通过web日志，定位webshell出现时间，查看webshell路径第一次出现前的web日志，判断攻击入口

              • 通过webshell所在目录用途，大致判断前台功能点，如webshell目录下其他文件均为合同文件，则前台可能是某上传合同的功能存在漏洞

            • SQL注入

              • 有流量设备

                • 从流量设备查看对应时间点的访问记录，可以找到入口

              • 无流量设备

                • 能查看对应web日志，如果为get型，可以直接从URL参数找到攻击点，如果为post型，则先定位命令执行的时间点，查看攻击发生前的web日志，当发现出现大量重复URL时，就是攻击入口

            • …

          • php等

            • 通过web的文件上传功能

            • SQL注入

            • …

          • asp、aspx等

            • 通过web的文件上传功能

            • SQL注入

            • …

      • 系统执行

      • …

可疑操作

  • 基于正则规则对攻击者常用手法进行的匹配判断，存在一定误报漏报

    • 确认为恶意操作

      • 通过审计日志对前后的操作进行审计，查看是否进行其他恶意行为

      • 着重关注登录用户、执行命令用户、实际执行命令进程、登录ip、执行命令进程

          - 攻击者通过ip为140的机器，使用eadp用户通过ssh登录到了14机器上执行了命令，说明eadp的认证信息已失陷

  - 排查登录日志，查看告警附近时间段内，140主机以及eadp用户还登陆过哪些机器，并对其他被害机器进行排查

    • 不确认是否为恶意

      • 例如有些运维也习惯执行history -c，或其他命令造成告警

        • 通过告警上下文执行的其他命令进行判断，是否有其他可疑操作，若还无法判断，则联系用户进行确认

web命令执行

  • 大部分是基于进程树调用关系来进行判断，从web中间件到系统命令会告警，还有部分反弹shell等恶意行为

    • 主要看后续的系统命令执行的内容是否为恶意，以及整个进程树的调用过程

      • 如果进程树父进程为php相关（php-fpm），或iis相关（w3wp.exe），后续恶意进程执行的为系统命令，攻击者大概率通过webshell上传或SQL注入，进一步执行的系统命令，需要进行webshell或漏洞排查

        • 与web后门排查类似

      • Java相关父进程，Tomcat、weblogic等

        • 通过组件或中间件rce漏洞直接执行

          • 通过风险发现中的漏洞检测功能做简单排查

        • 通过webshell执行

          • 任意文件上传

          • SQL注入

          • …

内存后门

  • 一般分为内存webshell和内存恶意代码两种

    • 内存webshell类

      • 看下告警详情，对于常见的内存后门一般都可以直接识别，主要还是需要下载Java源码下来进行分析，看下有没有执行系统命令之类的内容

        • 内存后门一般都为rce直接植入的，所以排查方式与上面web命令执行的排查方式基本一致，有些是为了隐蔽性，通过已有webshell进一步植入的内存马

    • 内存恶意代码

      • 这种和上面的类似，也是先看告警详情，看下是报的恶意工具，还是恶意shellcode，然后对sha256值进行分析，威胁情报平台没有的话可以考虑下载下来恶意文件进行上传分析

        • 一些其他厂商的edr之类会产生误报

        • 一般来说最外部也是通过一些rce漏洞打进来，进而进一步在内存里执行。如果是已经在内网，想要在另一台主机上直接内存执行的话，需要先获取权限，例如通过钓鱼之类，这就需要从别的方面进行检测。

异常文件监控

  • 主要是针对一些系统自带敏感文件的一些读写改的监控

    • 告警会比较多，监控时要多利用进程树，以及涉及的文件及操作，综合判断该行为是否为正常的业务行为，对于攻击行为的判断需要一定的经验，也需要结合业务场景去判断。如图中的一些进程必须读取某些文件的操作

    • 确认发生攻击时，需要结合审计日志以及其他告警进行排查

微蜜罐

  • 主要是为了捕获高级攻击者的隐蔽扫描行为

    • 发生告警时，研判方式与暴力破解类似，当出现同一源ip短时间内对不同ip的多个端口或一个端口进行扫描时，则有较大概率判断为攻击行为，对攻击ip进行排查即可

    • 有些运维工具也会产生类似的告警，如ansible

    • 有些有经验的攻击者不会进行大范围的扫描，只会手动精确的对某一台进行进行少量端口测试，如常见的80,443，这时候蜜罐可能无法发挥作用

  • 天翼云这边公网没开蜜罐，内网蜜罐如果出现外网访问，需要跟安全员进行确认

    • 是否因为更改了网络策略导致的主机能被外网访问到

    • 如果机器存在公网网卡，建议将该机器的蜜罐策略给关了，蜜罐端口会默认开在公网网卡上会导致直接将端口暴露在外边
